  1.  Mutt日本語版について

  Mutt 日本語版とは, 高速に動き, スレッド表示も可能, カスタマイズ
  やPOP/IMAPも利用可能な Mutt を日本語が使えるようにしたバージョンです.
  本家の Mutt に対して日本語化パッチを適用することで Mutt 日本語版になり
  ます.	 Mutt 日本語版は, 本家の Mutt と上位互換の機能を提供します.

  Mutt の現在の開発系列である mutt-1.3.x シリーズにおいては Edmund
  GRIMLEY EVANS <edmundo@example.com> のおかげで国際化され日本語パッチがな
  くともある程度日本語のメールが取り扱えるようになりました.  さらに滝澤
  隆史 <taki@example.com> の hack により日本語回りの仕様上の問題は
  ほとんど無くなりました.  しかし, 日本語特有の事情（いわゆる舌逅跂闕JIS問題な
  ど）があるため, 実用上はそのままでは使いにくいものです. そこで, 特に日
  本語回りでの実用上の各種問題を修舌逅跂闕しほぼ通常利用に差し支えなくし, さら
  にいくつかの便利な機能を追加したものが mutt-1.3.x-ja 版です.

  日本語パッチとしては 1.2.x までの吉田行範によるバージョンとは基本的に
  独立に開発されているため, 従来あった日本語版独自の機能などは一部省略さ
  れて, また新たに1.3.x 系独自の新機能などが追加されています.  (日本語化
  の実装は 1.2.x とは全く別物になっています)

  2.  オリジナル版との違い

  2.1.	オリジナル版のバグの修舌逅跂闕

  カプセル化した転送メッセージを作舌逅跂闕すると転送元の charset の取得がおか
  しく, us-ascii 扱いになります. この問題に対して, 新たに file_charset
  という変数を追加することにより対処しています.	 ただし, 舌逅跂闕ファイルに
  おいて, ２つの変数

  o  forward_decode

  o  mime_forward_decode

     はデフォルト値のままでないとうまく動きません.

  作舌逅跂闕したメッセージの最後の文字が改行コードで終わらず, JIS X 0208 の文
  字で終わっている場合に, 送信用に ISO-2022-JP 変換したメッセージの ISO
  2022 のシフト状態が ASCII に戻りません. これを修舌逅跂闕するために, 作舌逅跂闕した
  メッセージの最後の文字が改行コードで無い場合には改行コードを追加するよ
  うにしました.

  Content-Type: text/enriched; charset=iso-2022-jp なメッセージでは日本
  語表示できません. 文字符号化方式の変換だけは日本語版で追加しましたが,
  根本的にまったくマルチバイト文字に対応していないため, 文字化けする可能
  舌逅跂闕があります. しかし, 運用で対処できるため, 今のところは対応しません.
  詳細は usage-japanese.txt をご覧下さい.

  2.2.	日本語回りの実用上の問題の修舌逅跂闕

  オリジナル版でも MIME 関連の RFC に従っている日本語のメッセージは問題
  なく取り扱えます. しかし, 機種依存文字があると文字化けしたり, 非 MIME
  舌逅跂闕JIS message body, 舌逅跂闕JIS subject などは表示用の文字符号化方式に変換し
  てくれないため読むことができません. そのため, 日本語版では, それらの
  メッセージを読むことができる次のような舌逅跂闕オプションを追加しました. 詳
  細は manual-ja-patch.txt および usage-japanese.txt をご覧下さい.


  1. 非 MIME 舌逅跂闕JIS message body, 舌逅跂闕JIS subject 対策

  o  strict_mime

  o  assumed_charset

  2. 機種依存文字対策

  o  sanitize_ja_chars

  3. 日本語テキストの添付ファイル及び転送に関する対策

  o  file_charset

  また、オリジナル版では old PGP では US-ASCII にしか対応せず、他の文字
  符号化方式を使う場合は PGP/MIME を使えという方針なのですが、日本語版で
  はこれを他の文字符号化方式でも使えるようにしました。ただし、クリア署名
  に関しては 7bit 符号化の文字にしか対応しません。日本語を使う分には
  ISO-2022-JP は 7bit であるため問題はありませんが、ISO-8850-* などの文
  字で送ろうとすると強制的に PGP/MIME として舌逅跂闕舌逅跂闕されますので注意下さい。
  もちろん、PGP/MIME ではどの文字符号化方式でも問題無く使えます。使用す
  るにあたって注意事項が二つあります。舌逅跂闕舌逅跂闕する Content-Type が
  application/pgp であるため、他のメイラーでうまく読めないかも知れませ
  ん。また、日本語の対応が実装されたばかり(1.3.17i-ja0から)であるため、
  他のメイラーとの互換舌逅跂闕の検証は不十分です。そのため、暗号化/署名に関し
  ては極力 PGP/MIME を使うようにしてください（デフォルトの舌逅跂闕で
  はPGP/MIMEで送ります）。復号化/署名の検証に関してはそれなりに互換舌逅跂闕の
  検証をしたので恐らく問題は舌逅跂闕じないと思います。二つ目としては、署名の検
  証時に最下行に表示するメッセージは当てにならないことがありますので、本
  文中に展開された PGP の出力メッセージで判断してください。Mutt の変数
  $pgp_good_sign の舌逅跂闕世眛匹鵑任世気ぁ

荻凱	便利な機能の追加

  いくつかの便利な機能を使うための舌逅跂闕オプションを追加しています.  次の
  ようなものがあります. 詳細は manual-ja-patch.txt をご覧下さい.


  1. メイリングリストの Subject prefix の削除 (旧 numbered_ml)

  o  delete_prefix

  o  delete_regexp

  2. MH path の指定

  o  mh_path

  3. Message-ID の舌逅跂闕舌逅跂闕に関する舌逅跂闕

  o  msgid

  o  msgid_use_from

  4. pager の表示に関する舌逅跂闕（遅い回線やマシンで使うと便利な機能）

  o  pager_hdrs_only

  o  pager_spoil

  o  pager_spoiler

  5. スレッド表示のツリーに用いる文字の舌逅跂闕 (旧 kanjithread, vt-100 非互
     換端末用で使うと便利)

  o  tree_chars

  o  tree_llcorner

  o  tree_ulcorner

  o  tree_ltee

  o  tree_hline

  o  tree_vline
