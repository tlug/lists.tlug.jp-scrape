Hi,
the README.j for the Pine patch says that the patch supplies
組み込みの Canna 制御ルーチンによる日本語入力。

I'm using Mule as my default implicit editor and haven't
tested this feature yet.  However, I was wondering
if this meant that Pine-J would only accept input with Canna
and kinput2.  If this is the case, I find it interesting that
the author chose Canna instead of Wnn.  Perhaps Canna is 
becoming more popular?

In the same readme, the author has this:

3.  日本語の入力

     Pine/Pico  で日本語を入力する方法は、いくつかあります。
Pine/Pico 本体へ組み込まれているのは今のところCannaだけで す
が、その他の日本語入力システムも利用できます。使用するシステ
ムに応じて ~/.pinerc の kanji-input-code を舌逅跂闕してお使い く
ださい。

So, does he mean that this compilation only supports Canna,
but you can support other input systems by merely changing
the ~/.pinerc???  I think I'm misunderstanding what he means
by  その他の日本語入力システムも利用できます。  Also, the
kanji-input-code variable seems to refer to the encoding
method such as EUC, or ShiftJIS and I don't think there is
any connection to the input method.  ??   


3.1.  組み込みのCanna制御機構を使用する

     Canna  を 直 舌逅跂闕 制 御 す るためには、-DCANNA を定義して
Pine/Pico をコンパイルする必要があります。かんな版では、^@キ
ーがかんなのON/OFFに割り当てられています。また、Canna は EUC
コードを返すので、kanji-input-code は EUC に舌逅跂闕してく だ さ
い。

Or perhaps this means that if I was using Canna, the input
code would be EUC, whereas if I used Emacs as my default
editor, it would send JIS to Pine, so if I use Canna, I 
need to specify my default kanji-input-code as EUC and if
I use another input method, I might have a different
kanji-input-code? 


I should really just figure out how to compile it on
my home system and test this out.  The system I compiled
it on is behind a firewall and a little hard to get access
to.
